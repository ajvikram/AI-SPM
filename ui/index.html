📖 User Guide

Complete guide to AI-SPM Security Posture Management

📑 Table of Contents

1. Introduction 2. Auto Discovery 3. Dependency Graph 4. Agent Registration 5. Tokens & JIT Credentials 6. Real-Time Monitoring 7. Security Controls 8. Network Sentinel 9. Compliance & Reporting 10. Configuration 11. API Reference

1. Introduction

AI-SPM (AI Security Posture Management) provides end-to-end visibility and control over AI agents, LLMs, MCP servers, and IDE extensions running in your environment.

AI-SPM treats every agent as an untrusted principal within a deterministic control plane — enforcing identity, policy, taint tracking, tamper-evident auditing, and continuous monitoring. Built on NIST AI RMF, MITRE ATLAS, OWASP Agentic Top 10, and CSA MAESTRO frameworks.

Getting started: Launch the desktop monitor with cargo run --bin ai-spm-monitor. The embedded server starts automatically on port 8080 and the dashboard opens in the system tray.

2. Auto Discovery

The Auto Discovery engine scans your machine every 10 seconds and detects three categories of AI components:

🤖 AI Processes Ollama, Cursor, VS Code, Claude Desktop, Aider, LM Studio, Tabby, Cody, and more — detected by scanning running processes.
🔌 MCP Servers Parses claude_desktop_config.json, mcp.json, and 5 other known config locations. Extracts server name, command, transport, and environment variables.
🧩 IDE Extensions Scans VS Code, Cursor, Antigravity, and VS Code Insiders for 16 known AI extensions (Copilot, Claude, Cody, Codeium, etc.).

Navigate to Auto Discovery → click Register on any item to assign a SPIFFE identity, or use ⚡ Register All for bulk onboarding.

3. Dependency Graph

The Dependency Graph visualizes how agents, MCP servers, and extensions are interconnected. It answers the question: "If this component fails or is compromised, what else is affected?"

  • Nodes represent discovered agents (🔵 blue), MCP servers (🟣 purple), and extensions (🟢 green).
  • Edges are automatically inferred from IDE ecosystem matching — e.g., Cursor agent is linked to Cursor extensions and Cursor MCP configs.
  • Click any node to highlight its blast radius — all connected nodes turn 🔴 red, showing the potential impact zone.
  • Hover over a node for a tooltip with version, status, transport type, and impact level.

The Blast Radius Analysis table below the graph ranks every node by the number of affected nodes, helping prioritize security hardening.

4. Agent Registration

Registered agents receive a SPIFFE ID (e.g., spiffe://local/agent/cursor) — a cryptographically verifiable identity. Registration options:

  • Automatic — One-click from Auto Discovery page
  • Manual — Use the "Register Agent" form in the Agent Registry
  • Batch — Click "⚡ Register All" on the Discovery page to register everything in one action
  • API — POST /api/agents with JSON body

Each registered agent can be suspended (temporarily disabled) or revoked (permanently deactivated) from the Agent Registry.

5. Tokens & JIT Credentials

AI-SPM uses Just-In-Time (JIT) scoped tokens instead of persistent API keys. Tokens are:

  • Time-bound — Expire after a configurable TTL (default: 5 minutes)
  • Scoped — Each token has explicit permissions (e.g., read:customers, write:orders)
  • Single-use — Cannot be replayed after consumption
  • Signed — HMAC-SHA256 signed to prevent tampering

Issue tokens via the Tokens page or POST /api/tokens/issue.

6. Real-Time Monitoring

The Desktop Monitor captures real-time telemetry from four watchers:

🖥️ Shell Commands Every command executed by AI agents. Each is classified as ALLOW, FLAG, or BLOCK based on policy.
📁 File Operations Create, modify, delete events with automatic sensitivity classification (normal, config, credential, system).
⚙️ Process Spawns New processes with suspicious pattern detection (e.g., reverse shells, crypto miners).
🌐 Network Connections Outbound and inbound connections with remote addresses. Suspicious destinations are flagged.

All events appear on the Agent Monitor page in real-time.

7. Security Controls

AI-SPM provides five layers of defense-in-depth security:

🚪 Gateway (OPA) Policy-based evaluation of agent intents using Open Policy Agent. Write custom Rego policies and test them from the OPA Policies page.
🔧 MCP Tester Validates and sanitizes Model Context Protocol tool calls. Detects hidden instructions, SSRF payloads, and excessive permissions.
🛡️ Shell Guard Classifies shell commands as ALLOW / FLAG / BLOCK. Detects dangerous patterns like rm -rf, curl | bash, and privilege escalation.
📂 File Sentinel Controls file system access by sensitivity. Blocks writes to SSH keys, cloud credentials, and system configs.
🌐 Network Sentinel Evaluates outbound domains against allowlists, blocklists, and suspicious TLD patterns. Prevents data exfiltration.

8. Network Sentinel

The Network Sentinel evaluates outbound network requests made by agents. It works in two modes:

  • Domain Evaluation — Checks domains against known allowlists (e.g., api.github.com), blocklists, and suspicious TLD patterns.
  • Connection Monitoring — The monitor watcher passively observes all outbound TCP connections and flags suspicious destinations.

Test domain evaluation via the Network Sentinel page or POST /api/network/evaluate.

9. Compliance & Reporting

The Reports page maps AI-SPM capabilities to major security standards:

  • OWASP Top 10 for LLM Applications (2025)
  • NIST AI Risk Management Framework
  • MITRE ATLAS
  • ISO/IEC 42001
  • EU AI Act (Regulation 2024/1689)

Export compliance reports as HTML, CSV, or PDF from the Reports page.

10. Configuration

All settings are editable from the Configuration page:

  • Server — Host, port, database path, audit log location
  • Identity — SVID TTL, token signing key, default token TTL
  • Gateway — Nonce cache size, policies directory
  • Audit — Log file, index database, genesis hash

Configuration is stored at ~/.ai-spm/monitor.toml and auto-detects OS-appropriate paths.
